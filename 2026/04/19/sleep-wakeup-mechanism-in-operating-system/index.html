sleep-wakeup mechanism in operating system

发表于 2026-04-19

自旋锁可以解决短时资源竞争问题，但对于长时或不确定性的资源竞争使用自旋锁来解决便不合适了。一个典型的例子是磁盘读写：假设如下

  1. 进程期望向磁盘写入一系列字符，
  2. 但磁盘每次只能接受一个字符，
  3. 当磁盘将接受到的字符持久化后，通过中断的方式通知进程可以继续发送下个字符

在进程已经发送一个字符后、中断来临前的这段时间内是磁盘完成持久化所需的时间，而这经过的时间往往能让 CPU 运行上千条指令（或许更多？）。因此，在这段时间内持有锁并不断轮询不是一个好的主意。

Sleep-wakeup 机制是解决该问题的一个方法，他通过协作式的方式来调节进程的运行状态。接下来我们将介绍如何设计这种机制。在这篇文章中，我们会先给出一个简单、但有问题的实现，然后逐步纠正至正确的实现。

一种可能的解决方案

我们的目标是使同一个 event 的 wakeup 一定发生在 sleep 进入阻塞态之后。

核心思路是：若 sleep() 接受条件锁作为参数，则内部可依次：

  1. 获取进程锁 p.lock
  2. 释放条件锁 lock
  3. 将进程状态置为 SLEEPING

由于 wakeup() 同样需要 p.lock 才能修改进程状态，在步骤 1 完成后，wakeup() 便无法介入，直到进程完全进入阻塞态并释放 p.lock 后才能执行。这样便消除了"检查条件"与"进入阻塞态"之间的竞态窗口。具体实现将在后续章节中给出。

sleep-wakeup 具体实现

现在我们来到了最关键的部分。前面我们提到： > sleep() 会让进程进入阻塞态，而 wakeup() 则会将唤醒正在等待 event 到来的进程，使其从 sleep() 后继续运行

那这样的 sleep() 和 wakeup() 具体该怎么实现呢？

  1. 从当前进程的内核态切换至调度器线程
  2. 从调度器线程切换至被调度的进程的内核态

swtch 通常是配对使用的，也就是说第一种情况出现后，马上会出现第二种情况。通常这意味着进程放弃了 CPU 的执行权，出让给其他进程。

这里有一个关键的问题，锁一直持续到 swtch 执行完还存在？！是的，如果我们在 swtch 前释放，那在 swtch 执行过程中，其他 CPU 便有可能重新调度此进程，造成不一致的状态。 解决的办法就是在 swtch 后的进程自己释放锁，一个具体的例子就是 sleep()。不过在介绍 sleep() 前，先来看看更简单的 wakeup()。

这种"一方加锁、另一方解锁"的模式称为锁转移（lock handoff）。调度器与 sleep() 形成如下镜像关系：

  • 调度器 acquire(p.lock) → swtch 切换至进程 → 进程（即 sleep() 内）负责 release(p.lock)
  • sleep() acquire(p.lock) → swtch 切换至调度器 → 调度器负责 release(p.lock)

因此，虽然两者看起来都在"加锁 → swtch → 解锁"，但实际上互为镜像，p.lock 在任何时刻都不会被双重持有。

Wakeup 的实现

值得注意的是，wakeup() 采用的是广播语义：它会唤醒所有等待同一 event 的进程，而不只是其中一个。被唤醒的进程重新获得 CPU 时，条件可能已经再次不满足（例如另一个进程抢先消费了资源）。这就是为什么调用 sleep() 的外层应始终使用 while 而非 if 来检查条件——if 无法应对被误唤醒或被多个进程竞争的情形。

总结

至此，我们完整实现了 sleep-wakeup 机制。其设计核心在于：

  • 通过进程锁 p.lock 消除"检查条件"与"进入阻塞态"之间的竞态窗口
  • 通过锁转移约定，使调度器与进程在 swtch 前后安全地传递锁的所有权
  • wakeup() 采用广播语义，因此调用方必须在循环中检查条件

这实际上正是条件变量（condition variable）的底层实现原理：sleep() 对应 cv.wait(lock)，wakeup() 对应 cv.notify_all()，而传入的 lock 对应条件变量绑定的互斥锁。
