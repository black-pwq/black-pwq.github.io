sleep-wakeup mechanism in operating system

发表于 2026-04-19

自旋锁可以解决短时资源竞争问题，但对于长时或不确定性的资源竞争使用自旋锁来解决便不合适了。一个典型的例子是磁盘读写：假设如下

  1. 进程期望向磁盘写入一系列字符，
  2. 但磁盘每次只能接受一个字符，
  3. 当磁盘将接受到的字符持久化后，通过中断的方式通知进程可以继续发送下个字符

在进程已经发送一个字符后、中断来临前的这段时间内是磁盘完成持久化所需的时间，而这经过的时间往往能让 CPU 运行上千条指令（或许更多？）。因此，在这段时间内持有锁并不断轮询不是一个好的主意。

Sleep-wakeup 机制是解决该问题的一个方法，他通过协作式的方式来调节进程的运行状态。接下来我们将介绍如何设计这种机制。在这篇文章中，我们会先给出一个简单、但有问题的实现，然后逐步纠正至正确的实现。

一种可能的解决方案

我们的目标是使同一个 event 的 wakeup 一定发生在 sleep 进入阻塞态之后。

核心思路是：若 sleep() 接受条件锁作为参数，则内部可依次：

  1. 获取进程锁 p.lock
  2. 释放条件锁 lock
  3. 将进程状态置为 SLEEPING

由于 wakeup() 同样需要 p.lock 才能修改进程状态，在步骤 1 完成后，wakeup() 便无法介入，直到进程完全进入阻塞态并释放 p.lock 后才能执行。这样便消除了"检查条件"与"进入阻塞态"之间的竞态窗口。具体实现将在后续章节中给出。

sleep-wakeup 具体实现

现在我们来到了最关键的部分。前面我们提到： > sleep() 会让进程进入阻塞态，而 wakeup() 则会将唤醒正在等待 event 到来的进程，使其从 sleep() 后继续运行

那这样的 sleep() 和 wakeup() 具体该怎么实现呢？

进程状态与进程切换

要解决的第一个问题就是如何实现进程阻塞和可运行的状态？一种简单的实现方式便是在 PCB 中维护一个 state 变量，并交由内核态的调度器线程来根据该变量的值来调度进程。

通过保存当前进程的上下文信息，并恢复调度器线程的上下文信息，最后跳转至调度器线程的指令地址便可以完成上下文切换。由于这些过程需要操作寄存器，所以我们使用 swtch() 函数来封装这一切:

通常，在两种语境下会调用 swtch():

  1. 从当前进程的内核态切换至调度器线程
  2. 从调度器线程切换至被调度的进程的内核态

swtch 通常是配对使用的，也就是说第一种情况出现后，马上会出现第二种情况。通常这意味着进程放弃了 CPU 的执行权，出让给其他进程。

上述代码实现了一个简单的调度器，但竞态问题仍然存在，其他 CPU 可能会在同一个进程上进行调度，所以我们需要锁来保护临界区：

这里有一个关键的问题，锁一直持续到 swtch 执行完还存在？！是的，如果我们在 swtch 前释放，那在 swtch 执行过程中，其他 CPU 便有可能重新调度此进程，造成不一致的状态。 解决的办法就是在 swtch 后的进程自己释放锁，一个具体的例子就是 sleep()。不过在介绍 sleep() 前，先来看看更简单的 wakeup()。

这种"一方加锁、另一方解锁"的模式称为锁转移（lock handoff）。调度器与 sleep() 形成如下镜像关系：

  • 调度器 acquire(p.lock) → swtch 切换至进程 → 进程（即 sleep() 内）负责 release(p.lock)
  • sleep() acquire(p.lock) → swtch 切换至调度器 → 调度器负责 release(p.lock)

因此，虽然两者看起来都在"加锁 → swtch → 解锁"，但实际上互为镜像，p.lock 在任何时刻都不会被双重持有。

Wakeup 的实现

值得注意的是，wakeup() 采用的是广播语义：它会唤醒所有等待同一 event 的进程，而不只是其中一个。被唤醒的进程重新获得 CPU 时，条件可能已经再次不满足（例如另一个进程抢先消费了资源）。这就是为什么调用 sleep() 的外层应始终使用 while 而非 if 来检查条件——if 无法应对被误唤醒或被多个进程竞争的情形。

总结

至此，我们完整实现了 sleep-wakeup 机制。其设计核心在于：

  • 通过进程锁 p.lock 消除"检查条件"与"进入阻塞态"之间的竞态窗口
  • 通过锁转移约定，使调度器与进程在 swtch 前后安全地传递锁的所有权
  • wakeup() 采用广播语义，因此调用方必须在循环中检查条件

这实际上正是条件变量（condition variable）的底层实现原理：sleep() 对应 cv.wait(lock)，wakeup() 对应 cv.notify_all()，而传入的 lock 对应条件变量绑定的互斥锁。

%AppData%目录详解

%UserProfile%\AppData通常存放用户相关的数据，其下包含3个文件夹

  1. Roaming，由环境变量%AppData%表示其路径。是许多软件存放（通常与用户相关的）数据的路径。如名中Roaming所暗示的一样，该文件夹内的内容可通过网络同步到不同电脑，前提是在同一域中登入同一账户。该特性在Windows Vista中被详细介绍。
  2. Local，由环境变量%LocalAppData%表示其路径。与Roaming文件夹相似，但不会进行同步。
  3. LocalLow，此文件夹通常较为少用，与Local文件夹相似，但具备更低的集成度。

常见的软件安装位置

%LocalAppData%\Programs是用来存放单用户安装程序的目录。%LocalAppData%\Programs与%LocalAppData%\ProgramData两者不会出现在%AppData%中。

例如vs code的安装路径就在%LocalAppData%\Programs。Vs code官网对单用户安装与系统级别的安装也进行了说明。

By default, VS Code is installed under
C:\Users\{Username}\AppData\Local\Programs\Microsoft VS Code.

事实上，许多软件会滥用%AppData%目录。例如百度云安装的目录就在%AppData%中，但一般而言，该目录应当存放软件使用过程中产生的数据，而不是软件本身。

目前，Windows 10下软件安装的位置仍然很多样，但比较常见的安装位置通常分布在%ProgramFile%, %ProgramFiles%\WindowsApps以及%LocalAppData%\Programs下。安装在前两者的应用通常是系统层级的，而后者多为用户独有应用，其中WindowsApps通常为微软商店下载的应用位置，即便具备管理员权限也通常不能访问。

常见的软件数据存放位置

如前所述，%AppData%与%LocalAppData%中通常存放用户相关的软件数据。在两者下，常见的文件夹有Microsoft与Google等。例如微软Edge浏览器的用户数据通常存放在%LocalAppData%\Microsoft\Edge中。此外，由微软发布的非应用级的软件也有可能存放在Microsoft文件夹下。例如，vs code中的C/C++拓展的缓存目录就存放于此。

%UserProfile%目录

该目录通常被称作用户家目录，其路径通常为C:\Users\<username>。该目录通常存放一些应用的配置文件相关信息，其大部分配置文件采用类Unix的命名方式，以.作为前缀，尽管Windows并未对该类文件夹或文件赋予默认隐藏属性。一些非常常见的文件夹或文件包括：

  • .m2用于存放Maven仓库
  • .p2存放Eclipse插件(Bundle)
  • .ssh存放ssh配置文件相关
  • .vscode
  • .android
  • ...

Independence and Suggestion

发表于 2023-10-10 更新于 2023-10-16

Independence

Page 73, Problem 1.43.

The events \(A\) and \(B^c\) are independent if \(A\) and \(B\) are independent. \[\begin{aligned} P(B)P(A) + P(A\cap B^c) = P(A) = P(B)P(A) + P(B^c)P(A) \end{aligned} \] The first equality holds because \[\begin{aligned} P(A\cap B) + P(A\cap B^c) = P(A) = P(B)P(A) + P(A\cap B^c) \end{aligned}\]

Transitivity of Independence on set's complement

The events \(A^c\) and \(B^c\) are independent if \(A\) and \(B\) are independent. Apply the previous result on \(A\) and \(B\), then on \(B^c\) and \(A\).

Suggestion

Page 69, Problem 1.29.

Let \(A\) and \(B\) be events with \(P(A) > 0\) and \(P(B) > 0\). We say that an event \(B\) suggests an event \(A\) if \(P(A|B) > P(A)\), and does not suggest event \(A\) if \(P(A|B) < P(B)\).

Suggestion is mutual

\[\begin{aligned} P(A|B) > P(A) \iff P(B|A)> P(B) \end{aligned} \]

\[\begin{aligned} P(A|B) &> P(A)\\ P(A\cap B)&>P(A)P(B) \end{aligned}\] Notice the symmetry in the above equivalent formula.

Suggestion is complementary

\[\begin{aligned} P(A|B) > P(A) \iff P(A|B^c) < P(A) \label{eq:complement} \end{aligned}\]

\[\begin{aligned} P(B)P(A) + P(B^c)P(A) = P(A)= P(B)P(A | B) + P(B^c)P(A | B^c) \end{aligned}\] If \(P(A|B) > P(A)\), to make the above equality holds, \(P(A)\) must be less than \(P(A|B^c)\).

Notice [eq:complement] could be rewritten as \[\begin{aligned} P(A|B) > P(A) > P(A|B^c), \end{aligned}\] if \(A,B\) suggest each other. Thus, the following argument is always true \[\begin{aligned} P(A|B) > P(A|B^c) \iff P(A|B) > P(A) \end{aligned}\]

Suggestion is not only complement on the condition, but also on the conditional event, i.e., \[\begin{aligned} P(A|B) > P(A) \iff P(A^c|B) < P(A^c) \end{aligned}\]

\[\begin{aligned} P(A|B) + P(A^c|B) = 1= P(A) + P(A^c) \end{aligned}\]

Double complement suggestion have the same result as the original. In other words, \[\begin{aligned} P(A|B) > P(A) \iff P(A^c|B^c) > P(A^c) \end{aligned}\] This equivalence is trivial by the above arguments.
